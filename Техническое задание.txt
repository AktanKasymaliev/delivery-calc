Техническое задание
Проект: Админ-панель для управления калькулятором расчета доставки грузов
1. Цель
Реализовать административную панель, через которую менеджер может настраивать зависимости расчета стоимости доставки.
На клиентской стороне пользователь:
* выбирает город отправки;
* выбирает город доставки;
* вводит вес;
* вводит объем;
* получает итоговую стоимость;
* получает срок доставки;
* при необходимости видит цену за кг.
Администратор в админке должен иметь возможность без участия разработчика:
* задавать направления доставки;
* задавать тарифы;
* задавать зависимости цены от веса и/или объема;
* задавать маржу;
* задавать срок доставки;
* включать/выключать направления;
* редактировать параметры расчета.
2. Бизнес-логика
Сейчас на экране калькулятора есть следующие параметры:
* город отправки;
* город доставки;
* вес;
* объем;
* цена за кг;
* стоимость;
* срок доставки.
В расчете стоимости должны учитываться:
* направление доставки;
* вес;
* объем;
* тариф;
* маржа.
Важно: администратор должен сам управлять зависимостями расчета через админ-панель.
3. Основной сценарий работы клиента
3.1. Пользовательский сценарий
1. Пользователь открывает калькулятор.
2. Выбирает город отправки.
3. Выбирает город доставки.
4. Вводит вес.
5. Вводит объем.
6. Нажимает кнопку «Рассчитать».
7. Система определяет подходящую тарифную запись.
8. Система рассчитывает итоговую стоимость.
9. Система показывает:
   * цену за кг;
   * итоговую стоимость;
   * срок доставки.
4. Логика расчета
4.1. Общий принцип
Расчет должен строиться на основе тарифной конфигурации, заведенной в админке.
Базово система должна поддерживать такую модель:
Входные параметры:
* город_отправки
* город_доставки
* вес
* объем
Параметры из админки:
* базовый тариф за кг
* базовый тариф за м³
* минимальная стоимость
* коэффициенты / зависимости
* маржа
* срок доставки
Формула
Должна быть возможность настроить один из следующих вариантов расчета:
Вариант A. Расчет по весу
стоимость = вес × тариф_за_кг + маржа
Вариант B. Расчет по объему
стоимость = объем × тариф_за_м3 + маржа
Вариант C. Расчет по наибольшему значению
Если используются и вес, и объем:
* считаем стоимость по весу;
* считаем стоимость по объему;
* берем большее значение;
* прибавляем маржу.
Формула:
стоимость = max(вес × тариф_за_кг ; объем × тариф_за_м3) + маржа
Вариант D. Смешанный расчет
стоимость = (вес × тариф_за_кг) + (объем × тариф_за_м3) + маржа
4.2. Рекомендуемый вариант
Для гибкости лучше сразу предусмотреть в админке поле «Тип расчета», чтобы по каждому маршруту можно было выбрать:
* по весу;
* по объему;
* по большему значению;
* смешанный.
Это даст возможность менять модель расчета без доработки кода.
5. Что должно настраиваться в админке
5.1. Справочник городов
Администратор должен иметь возможность:
* создавать город;
* редактировать город;
* деактивировать город.
Поля:
* ID
* Название города
* Страна
* Статус (активен / неактивен)
5.2. Справочник маршрутов
Администратор должен иметь возможность задавать направление доставки.
Поля маршрута:
* ID
* Город отправки
* Город доставки
* Статус (активен / неактивен)
* Срок доставки от
* Срок доставки до
* Валюта
* Тип расчета
* Комментарий для администратора
Пример:
* Отправка: Гуанчжоу
* Доставка: Бишкек
* Срок: 18–20 дней
* Тип расчета: по весу
5.3. Тарифная сетка маршрута
Для каждого маршрута должна быть возможность задать тарифы и зависимости.
Поля:
* Маршрут
* Минимальный вес
* Максимальный вес
* Минимальный объем
* Максимальный объем
* Тариф за кг
* Тариф за м³
* Маржа
* Минимальная стоимость
* Фиксированная надбавка
* Коэффициент для объема
* Коэффициент для веса
* Статус
5.4. Поддержка диапазонов
Админка должна позволять задавать разные тарифы в зависимости от диапазонов.
Пример:
Для маршрута Гуанчжоу → Бишкек:
* 0–10 кг - 180 $/кг
* 10.01–50 кг - 170 $/кг
* 50.01–100 кг - 160 $/кг
Либо по объему:
* 0–1 м³ - 200 $
* 1.01–3 м³ - 180 $
* 3.01–10 м³ - 160 $
Система должна выбирать подходящую запись по введенным параметрам.
6. Правила работы расчета
6.1. Выбор тарифа
После ввода пользователем параметров система:
1. находит активный маршрут;
2. среди тарифных записей находит подходящий диапазон по весу и/или объему;
3. применяет формулу расчета;
4. добавляет маржу;
5. если итог меньше минимальной стоимости — подставляет минимальную стоимость;
6. возвращает результат на фронт.
6.2. Маржа
Маржа должна поддерживаться в 2 вариантах:
* фиксированная сумма;
* процент.
Поле в админке:
* тип маржи: фиксированная / процент
* значение маржи
Пример:
* фиксированная: 20 $
* процент: 12%
6.3. Минимальная стоимость
Если рассчитанная сумма меньше установленного порога, система должна показать минимальную стоимость.
Пример:
* расчет дал 95 $
* минимальная стоимость по маршруту = 120 $
* клиенту показывается 120 $
7. Требования к фронтенду
7.1. Форма калькулятора
Поля:
* Город отправки — dropdown
* Город доставки — dropdown
* Вес — числовое поле
* Объем — числовое поле
* Кнопка «Рассчитать»
После расчета выводить:
* Цена за кг
* Стоимость
* Срок доставки
7.2. Валидация
* вес > 0
* объем > 0
* обязательный выбор города отправки
* обязательный выбор города доставки
Если данные не заполнены:
* показывать сообщение об ошибке.
Примеры:
* «Выберите город отправки»
* «Выберите город доставки»
* «Введите вес»
* «Введите объем»
7.3. Ошибки для клиента
Если для выбранного направления нет тарифа:
* «Для данного направления расчет временно недоступен»
Если маршрут неактивен:
* «Маршрут временно недоступен»

* по городу отправки

9. API требования
9.1. Метод получения справочников
GET /calculator/cities
Возвращает список активных городов.
GET /calculator/routes
Возвращает активные маршруты.
9.2. Метод расчета
POST /calculator/calculate
Request:
{
  "from_city_id": 1,
  "to_city_id": 2,
  "weight": 10.5,
  "volume": 1.2
}


Response:
{
  "success": true,
  "data": {
    "price_per_kg": 180,
    "total_price": 1890,
    "delivery_time_from": 18,
    "delivery_time_to": 20,
    "currency": "USD",
    "calculation_type": "weight"
  }
}


9.3. Ответ при ошибке
{
  "success": false,
  "message": "Для данного направления нет активного тарифа"
}

11. Логика выбора записи тарифа
Система должна:
* искать запись по активному маршруту;
* проверять попадание веса в диапазон;
* проверять попадание объема в диапазон;
* выбирать подходящую запись;
* если найдено несколько записей — применять самую точную или приоритетную.
Рекомендуется добавить поле:
* priority
Чтобы при пересечении условий система выбирала запись с большим приоритетом.
12. Нефункциональные требования
* Расчет должен выполняться не более чем за 2 секунды.
* Админка должна быть доступна только авторизованным пользователям.
* Все изменения тарифов должны логироваться.
* Система должна поддерживать масштабирование по количеству маршрутов.
* Значения должны храниться с точностью до 2 знаков после запятой.
* Валюта должна задаваться отдельно.
13. Права доступа
Роли:
Администратор
* полный доступ
* создание / редактирование / деактивация маршрутов, тарифов, городов
Менеджер
* просмотр
* редактирование тарифов без удаления
* без доступа к ролям и системным настройкам
Наблюдатель
* только просмотр
14. Acceptance Criteria
14.1. Для клиента
* клиент может выбрать город отправки и доставки;
* клиент может ввести вес и объем;
* после нажатия на кнопку получает итоговую стоимость;
* отображается срок доставки;
* если маршрута нет — выводится ошибка.
14.2. Для администратора
* можно создать маршрут;
* можно создать для него тариф;
* можно задать маржу;
* можно указать диапазоны веса и объема;
* можно изменить срок доставки;
* изменения начинают использоваться в калькуляторе без релиза фронта.
15. Пример бизнес-правила
Маршрут: Гуанчжоу → Бишкек
* Тип расчета: по весу
* Тариф за кг: 180 $
* Маржа: 10 $
* Срок: 18–20 дней
Если клиент ввел:
* вес = 1 кг
* объем = 1
Тогда:
* стоимость = 1 × 180 + 10 = 190 $
* срок = 18–20 дней
16. Что я бы рекомендовал добавить сразу
Чтобы потом не переделывать, лучше сразу заложить:
1. Черновик / публикация тарифа
Чтобы можно было сохранить тариф без мгновенного применения.
2. История версий
Чтобы видеть, какой тариф был до изменения.
3. Дата начала действия тарифа
Чтобы можно было заранее завести новые ставки.
4. Импорт тарифов через Excel
Если направлений станет много, руками это будет неудобно.
5. Отображение последней даты обновления тарифа
Полезно и для команды, и для контроля актуальности.
17. Краткая постановка для разработки
Нужно реализовать админ-панель для калькулятора доставки, в которой администратор сможет управлять:
* городами отправки и доставки;
* маршрутами;
* тарифными ставками;
* диапазонами веса и объема;
* маржой;
* сроками доставки;
* логикой расчета.
На фронтенде клиент вводит параметры и получает готовый расчет по данным из админки.
